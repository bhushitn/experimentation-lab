Switchbacks: testing when the marketplace is the unit

A delivery marketplace wants to A/B test a new dispatch algorithm. It cannot randomize couriers: both arms would draw from the same pool, and the treatment's effect on supply would bleed straight into control. So it randomizes time itself, flipping the whole city between algorithms in windows. Then the question becomes: how long is a window?

400 simulated 28-day city experiments per window length; true steady-state effect 1.0, with 40% of each hour's outcome carried from the previous hour and day-level market shocks. Reproduce this figure.

Randomize time, not users

Each colored band is a window where the whole market runs one arm. Within a window the marketplace equilibrates around that arm's algorithm; the analysis compares hours under treatment to hours under control. One market, one experiment, no user-level contamination.

Markets do not reset on the hour

Courier positions, queued orders, and surge state persist across the flip: in this simulation 40% of each hour's outcome is carried from the previous hour. The first hour of every control window still half-runs the treatment, and vice versa. The contamination is symmetric, so it does not cancel; it dilutes the contrast.

Short windows, big bias

The dilution is worst when windows are short, because a larger share of hours sit right after a flip: one-hour windows estimate 0.61 of a true 1.0, two-hour windows 0.79, day-long windows 0.97. The bias matches the exact carryover arithmetic, not just the simulation.

Long windows, big variance

So use day-long windows? Now each window rides one day's market shocks (weather, a promo, a concert) and the effective sample is the number of days, not hours: the standard error doubles from 0.21 at two-hour windows to 0.42 at day-long ones. This is the design tradeoff switchbacks force: bias on one side, variance on the other.

Burn-in buys out the bias

Drop the first hour after every flip and re-estimate: the burn-in analysis reads 0.99 at two-hour windows, on the truth, while keeping the variance of short windows. The practical design: windows a few multiples of the system's memory, plus a burn-in matched to it. Knowing your marketplace's decay time is the actual expertise.

The memo you write instead

Recommendation

Run the dispatch test as a switchback with 2-hour windows and a 1-hour burn-in after each flip, randomizing the window schedule. Simulated against a known truth with 40% hour-to-hour carryover: naive analysis of 2-hour windows understates the effect by 21%, while the burn-in estimate is unbiased (0.99 against a true 1.00) at half the standard error of day-long windows.
