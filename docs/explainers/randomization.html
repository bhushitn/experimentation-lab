What randomization actually buys you

A short-video app ships a new creation tool behind a toggle and, a month later, compares users who adopted it to users who did not. Adopters create twice as much. The tool is credited. The tool did a quarter of that.

Simulated population of 100,000 users with the true effect fixed at 0.50. Reproduce this figure.

Compare adopters to everyone else

The natural first analysis: mean outcome of users who adopted the tool minus mean outcome of users who did not. It reads 1.94. The true effect, set by construction in this simulation, is 0.50. The estimate is almost four times the truth.

The gap was there before launch

Adoption was not a coin flip: in this population the probability of adopting rises with a user's baseline activity, which is how real feature adoption behaves. Heavy users adopt more and also create more with or without the tool. That pre-existing gap, 1.43 here and computable exactly from the simulated population, rides into the comparison as if the tool caused it.

Flip a coin instead

Randomize the same users and the two groups have the same baseline mix by construction; nothing pre-existing can hide in the contrast. The randomized estimate is 0.47, noise away from the true 0.50. This is the entire purchase: assignment independent of everything about the user, known and unknown.

What it cannot fix

Randomization guarantees comparable groups at assignment. It does not guarantee the assignment is what users experience (contamination), that one user's treatment stays out of another user's outcome (network effects), or that the experimenter reads the result honestly (peeking). Those failures each get their own page and their own number.

The memo you write instead

Recommendation

Do not report the adopter-versus-non-adopter comparison as the feature's effect; in a simulated population with the truth fixed at 0.50, that comparison reads 1.94 because adoption correlates with baseline activity. Run the randomized experiment; it recovers 0.47 on the same population.
